## Context

当前扩展图标（store-128.png 及各尺寸变体）的设计内容未填满画布，深色背景方块周围存在透明像素，导致在 Chrome Web Store 和浏览器工具栏的浅色背景上出现可见的边缘伪影。

现有图标设计元素：深色背景 + 心电图脉冲波形 + Pac-Man 吃豆人造型，整体风格为像素风/复古风。

## Goals / Non-Goals

**Goals:**
- 生成边缘干净、完全填满画布的商店图标（128×128）
- 同步生成所有所需尺寸（16/32/48/96/128），保持视觉一致
- 保留现有的像素风/脉冲波形/Pac-Man 设计语言

**Non-Goals:**
- 不重新设计图标的核心创意概念（仍保留脉冲+吃豆人主题）
- 不修改 manifest.json 或图标引用路径
- 不添加自动化图标生成流水线

## Decisions

### 决策 1：使用 AI 图像生成重新创建图标

**选择**：使用 AI 图像生成工具（如 DALL-E / Gemini）根据现有设计风格重新生成 128×128 基础图标，确保内容完全填充画布。

**替代方案**：
- 手动用图像编辑工具修复透明边距 → 简单但可能引入新的边缘质量问题
- 使用 SVG 重绘后导出 → 工作量大，像素风格用 SVG 不自然

**理由**：AI 生成可以快速迭代，同时保持像素风格的一致性。生成后用脚本缩放到各尺寸。

### 决策 2：从 128px 基础图向下缩放生成各尺寸

**选择**：以 128×128 为基础，使用 ImageMagick 或 Sharp 等工具缩放生成 96/48/32/16 尺寸。

**理由**：保持所有尺寸视觉一致，避免分别生成导致的风格差异。

### 决策 3：图标背景处理

**选择**：图标使用圆角矩形深色背景，完全填充画布，无透明区域。

**替代方案**：使用全透明背景 + 图标内容居中 → 在不同背景色下效果不可控

**理由**：Chrome Web Store 推荐图标自带背景，避免依赖平台背景色。

## Risks / Trade-offs

- **AI 生成质量风险** → 可能需要多次迭代 prompt 才能得到满意结果；如果效果不佳，回退到手动修复透明边距的方案
- **缩放质量风险** → 16×16 等极小尺寸可能丢失细节 → 使用适合像素艺术的最近邻缩放算法，必要时对小尺寸单独微调
- **品牌一致性风险** → 新图标可能与用户已熟悉的旧图标差异较大 → 尽量保持相同配色和构图
